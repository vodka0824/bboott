# LINE Bot 專案開發規範 (Design Guidelines)

本專案為了提供穩定且不中斷的服務，並避免超過 LINE 官方帳號免費版的資源限制，訂定了以下核心開發規範。所有後續的模組開發與維護，都必須嚴格遵守這些規定。

## 1. 核心通訊規範：嚴禁使用主動推播 (Push Message)

### 🚫 禁止行為
- **絕對禁止**使用 LINE SDK 的 `pushMessage` API。
- **絕對禁止**在 `utils/line.js` 或任何地方呼叫 `lineUtils.pushMessage`。
- 不可依賴任何主動將訊息「塞」給使用者的機制。

### ✅ 解決方案：一律使用被動回覆 (Reply Message)
- 所有的訊息互動，必須建立在「使用者主動發送訊息」的前提下。
- 使用 `replyToken` 搭配 `context.replyToLine` 或 `context.replyFlex` 來回覆使用者。
- **注意**：一個 `replyToken` 只能被消耗**一次**，且有效期限為 1 分鐘。請確保在生命週期內完成回覆。

### 🔄 非同步與跨群組通知的處理 (信箱佇列機制)
如果遇到需要「主動通知」的情境（例如：背景定時任務觸發經濟事件、跨群組的全服廣播、超過一分鐘的非同步處理），**請遵守以下實作模式**：
1. 將該通知訊息存入資料庫（例如 `notifications` 集合），並標註目標接收者（`groupId` 或 `userId`）。
2. 在路由層的攔截器中，每次收到使用者的 Webhook 訊息時，檢查該群組/使用者是否有「待辦通知」。
3. 若有待辦通知，則使用該次 Webhook 挾帶的 `replyToken`，將「待辦通知」與「使用者的正常指令回覆」合併為一個 Array 一併發送出，藉此達到「偽主動推播」的效果。
